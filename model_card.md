# Model Card: Tuneweave

---

## 1. Model Name

**Tuneweave**

A cross-lingual, natural-language music recommender that takes free-text taste descriptions and returns five grounded song recommendations across Telugu, Hindi, and English catalogs, with conversational refinement over up to three turns.

---

## 2. Goal / Task

Tuneweave answers one question: given what a user *describes* wanting to listen to, which songs from a multilingual catalog are the best match, and why?

It handles natural-language input ("something chill for late-night studying, maybe Telugu"), optional follow-up corrections ("not the first one, something more acoustic"), and explains each pick in 2 to 3 sentences grounded strictly in the song's metadata description. It does not learn from listening behavior over time; every recommendation is based on the current session's parsed preferences.

---

## 3. Algorithm Summary

Tuneweave is a five-stage pipeline:

**Stage 1, Preference Parsing (LLM):** The user's query is sent to Gemini 2.5 Flash-Lite with a structured output schema. The model extracts languages, mood descriptors, energy level (0 to 1 scale), listening context, and whether the language preference is strict or soft. Output is validated against a Pydantic schema; malformed responses retry once before falling back to a minimal profile.

**Stage 2, Semantic Retrieval (sentence-transformers):** The parsed query is embedded using `all-MiniLM-L6-v2`. Cosine similarity is computed against precomputed embeddings of all 96 song descriptions. Hard filters (strict language, excluded IDs from prior turns) are applied. The top 40 candidates by cosine similarity are returned.

**Stage 3, Audio-Feature Ranking (rule-based):** The 40 candidates are scored by combining retrieval score (60% weight) with an audio-feature fit score (40% weight). Feature scoring uses energy proximity, explicit emphasis overrides from refinement turns, and a language soft bonus for single-language preferences. When multiple languages are requested, results are round-robin interleaved by language so no single language dominates the top 5.

**Stage 4, Batched Explanation (LLM):** All five ranked songs are sent to Gemini in a single call. The model generates a 2 to 3 sentence explanation for each, constrained to the retrieved song description and the user's stated preferences, with no invented facts and no quoted lyrics.

**Stage 5, Session Loop:** Results are displayed and the user can refine up to two more times. Refinement feedback is parsed by a second LLM call that produces a structured delta (exclusions, emphasis shifts, language changes) merged into the existing session profile.

---

## 4. Data Used

**Catalog:** 96 songs stored in `data/songs.csv`.

| Language | Count |
|---|---|
| Telugu | 50 |
| English | 28 |
| Hindi | 18 |

Each song has: title, artist, language, genre, mood tags, energy (0 to 1), tempo (BPM), valence (0 to 1), danceability (0 to 1), acousticness (0 to 1), year, and a 2 to 3 sentence description.

**Song descriptions** were generated by Gemini from structured metadata (genre, mood tags, audio features) with an explicit instruction not to invent biographical facts and not to quote lyrics. All 96 descriptions were manually reviewed; incorrect ones, particularly for Telugu and Hindi songs where Gemini's knowledge is thinner, were rewritten by hand.

**Embeddings** are precomputed using `sentence-transformers/all-MiniLM-L6-v2` and saved to `data/embeddings.pkl`. The embedding text for each song is prefixed with its language and genre to help the model distinguish cross-lingual tracks with similar descriptions.

**What's missing:** No songs below energy 0.10. Telugu songs are over-represented (52% of catalog). No explicit lyric content, artist popularity signals, or user listening history. Audio features are estimated from metadata rather than extracted from audio.

---

## 5. Strengths

**Cross-lingual matching works.** A query in plain English ("chill songs for studying") correctly surfaces Telugu songs by semantic similarity between the query and the Telugu song's English-language description. Users do not need to know the language of a song to discover it.

**Grounded explanations don't hallucinate.** The explainer is constrained strictly to the retrieved song description. In 15 manual evaluation sessions, no explanation contained invented biographical details, fabricated awards, or quoted lyrics. The fallback (returning the raw description) is itself grounded.

**Refinement meaningfully changes results.** Excluding a song by position ("not the second one"), shifting audio emphasis ("more acoustic"), and changing language preference all produce detectably different recommendation sets in subsequent turns.

**Batched explanation generation reduces LLM cost without quality loss.** A single call generates all five per-song explanations as a structured array, dropping per-turn LLM usage from 6 calls to 2. Sharing the call context also makes the five explanations more consistent in tone than independent calls would produce.

**Reliability guardrails work.** Every LLM call validates output against a Pydantic schema, retries once on failure, falls back to a safe default, logs the full exception to stderr, and is capped at a 30-second timeout. No failure mode silently produces garbage output.

---

## 6. Limitations and Bias

**Catalog skew toward Telugu.** With 50 of 96 songs, Telugu content has nearly twice the representation of English (28) and nearly three times that of Hindi (18). Users querying without a language preference may consistently receive Telugu-heavy results unless the language diversifier is triggered. Hindi users, with the smallest pool, have fewer options at every ranking stage.

**Embedding model is English-dominant.** `all-MiniLM-L6-v2` was trained primarily on English text. Even though song descriptions are written in English, the semantic meaning of Telugu and Hindi music concepts (raaga, shrungara, gaana) may not map cleanly into the embedding space the way Western genre vocabulary does. This means cross-lingual retrieval is less reliable for non-English musical vocabulary in queries.

**Gemini's knowledge of Telugu and Hindi music is uneven.** LLM-generated descriptions for Telugu and Hindi songs had a higher error rate than English songs; artists were sometimes confused, eras misidentified, or sonic qualities described inaccurately. Manual review corrected these, but the underlying asymmetry means auto-scaling the catalog via LLM generation would require proportionally more review time for non-English content.

**No signal beyond the current session.** The system cannot learn that a user consistently skips slow songs, prefers certain artists, or has changed their mood mid-session beyond what they explicitly say. Two users with identical queries get identical recommendations.

**Energy-as-proxy is imprecise.** Energy is a single continuous value estimated from metadata. "High energy" means different things in Telugu film music versus electronic pop, but the ranker treats the same numeric value identically across genres.

**Hard filter is binary.** When `language_strict=true`, any song not in the requested language is excluded entirely, even if it is semantically almost identical to what the user wants. There is no graceful degradation for strict filters that match very few songs.

## 7. Evaluation

**Automated tests:** 42 unit tests pass covering catalog loading, retrieval (hard filter correctness, K-size, exclusion handling), ranking (scoring monotonicity, language bonus logic, diversification), parser schema validation on cached LLM outputs, and session state transitions.

**Manual evaluation across 15 queries** spanning all three languages, multiple moods, strict and soft language preferences, and multi-turn refinement:

| Metric | Result |
|---|---|
| Preference parser, correct structured output | 14 / 15 |
| Recommendations, on-topic for query | 13 / 15 |
| Explanations, grounded, no invented facts | 15 / 15 |
| Refinement, result set changed meaningfully | 5 / 5 tested |

The one parse failure was an ambiguous query ("nenu vintunna padaalu", Telugu for "songs I'm listening to") where the parser returned an empty language list instead of inferring Telugu from context. The two off-topic recommendations occurred when an explicit acousticness emphasis from refinement conflicted with a high-energy hint from Turn 1, and the ranker weighted the emphasis override so heavily that it surfaced low-energy songs for a user who originally wanted high energy.

## 8. Intended and Non-Intended Use

**Intended for:**
* Personal music discovery across Telugu, Hindi, and English catalogs
* Demonstrating how retrieval-augmented generation, structured LLM output, and rule-based ranking can be composed into a working pipeline
* Educational exploration of applied AI system design

**Not intended for:**
* Real production music services; the catalog is 96 songs and coverage is too narrow
* Recommending music in languages outside Telugu, Hindi, and English
* Contexts where LLM hallucination in explanations could cause harm; this system is low-stakes but explanations are only as accurate as their source descriptions

**Potential misuse and mitigations:**
The primary attack surface is prompt injection through the user query field; a crafted input could attempt to override the system prompt and produce off-topic or harmful text. Two mitigations are in place: input is truncated at 500 characters before being sent to any LLM call, and all LLM outputs that return data (parser, explainer) are validated against fixed Pydantic schemas. A response that doesn't match the schema is rejected and retried; it cannot inject arbitrary content into the downstream recommendation or explanation. The explainer uses unstructured text output, which is more vulnerable; a future mitigation would be to run explanations through a grounding check before display.

## 9. Reflection

### Limitations and biases

The most concrete bias in the system is the catalog skew. 52% Telugu songs mean that a language-agnostic query almost always produces a Telugu-heavy result unless the round-robin diversifier is explicitly triggered by a multi-language request. This is not a model bias in the traditional sense; it's a data collection decision that produced unequal representation. Fixing it properly would require expanding the Hindi catalog from 18 to at least 30 songs, not tuning a parameter.

The subtler bias is in the embedding space. The `all-MiniLM-L6-v2` model represents music concepts through English vocabulary, and the semantic neighborhood of "melancholic Telugu film ballad" is probably closer to "sad English pop" than to other Telugu film ballads, because the model learned those relationships from English text, not from musical structure. Two Telugu songs that sound nearly identical might be placed far apart in embedding space if their descriptions use different English vocabulary.

### Misuse potential

A music recommender is low-stakes by nature; the worst likely misuse is a user trying to extract content the system wasn't designed to produce by crafting an adversarial query. The 500-character input cap and Pydantic schema validation on all structured outputs prevent prompt injection from affecting the parser's output. The explainer is the weakest link, since it produces free text, but its system prompt explicitly constrains it to the provided song description. A user who injects a fake "description" field has no pathway to do so; descriptions come from the catalog CSV, not from user input.

### Surprises during reliability testing

The most surprising failure was how completely silent the error swallowing was before adding proper logging. The system was running full sessions, printing results, and looking correct, while every single explainer call was failing with a 429 quota error and falling back to the raw `song.description`. The output was plausible enough that the failure was invisible until stderr logging was added and the traceback appeared. This taught a concrete lesson: a system that always produces output is not the same as a system that always works.

The second surprise was the language soft bonus bug. Adding a 0.3 bonus to songs in the user's preferred language is a reasonable design for single-language preferences. But when the user requests all three catalog languages (Telugu, Hindi, and English), every song in the 96-song catalog matches at least one of them, so every song gets the identical bonus. The bonus added a constant to all scores and changed nothing. This is the kind of edge case that is obvious in retrospect but requires actually running the system with a multi-language query to observe.

### AI collaboration

**Helpful suggestion:** When diagnosing the 5-calls-per-turn quota problem on the free-tier Gemini model, Claude Code suggested replacing the five individual explainer calls with a single batched LLM call that generates all five explanations at once and returns them as a structured JSON array. This reduced per-turn LLM usage from 6 calls to 2 (1 parser + 1 batch explainer), cutting the session quota footprint by 67% and making the system practically usable for iterating without hitting the daily limit. The suggestion also improved response consistency; all five explanations in a session share the same call context, so they're more coherent in tone.

**Flawed suggestion:** Earlier in development, after a self-audit of the codebase, Claude Code produced a list of bugs to fix, including "refinement loses original query intent" and "retry is logged as a failed call with no success log." It then proceeded to implement fixes for each item. The fixes were technically correct but addressed symptoms rather than the actual problem: every LLM call was silently failing because the 5-explanations-per-turn pattern was burning through Gemini's free-tier daily quota in a few sessions. The "broken refinement" was real, but it wasn't a logic bug; it was that all calls were returning errors and falling back to unchanged state. Claude Code was too eager to act on the stated symptom list rather than asking "why is the system actually behaving this way?" The lesson was a meta-lesson about working with AI assistants: stated problems are not always root causes, and an assistant that fixes what it's told to fix without diagnosing first will produce confident, plausible, irrelevant patches. The actual fix (switch model, batch the calls) only became visible after the symptom-level fixes failed and the underlying quota error was finally surfaced via stderr logging.
